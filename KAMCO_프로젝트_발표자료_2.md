### 📌 이 프로젝트가 필요했던 이유

#### ❌ 기존 KAMCO 사이트의 문제점

- 간단한 필터링 (지역, 가격대만)

| 데이터 | 165개 실제 경매 물건 |

## 🎪 SLIDE 3: 기술 스택

### 🛠️ 기술 스택 (왜 선택했나?)

#### Frontend (사용자 화면)
```
├─ HTML/CSS: 깔끔한 UI 디자인
└─ JavaScript: 실시간 상호작용
```

#### Backend (데이터 처리)
```
├─ Spring Boot: ⭐ 자바 웹앱 최고의 선택
│  └─ Spring Security: 안전한 인증
├─ MyBatis: SQL 최적화 가능
└─ JWT: 토큰 기반 인증 (세션보다 가벼움)
```

#### Database (데이터 저장)
```
├─ MariaDB: MySQL과 호환, 안정적
└─ 인덱스 + 캐싱: 빠른 검색
```

#### Production
```
├─ Swagger: API 자동 문서화
├─ Caffeine Cache: 메모리 캐싱 (응답 10배 빨라짐)
└─ HikariCP: 효율적인 DB 연결 관리
```

#### 기술 선택 근거

| 기술 | 하는 일 | 왜 사용? |
|------|--------|---------|
| **Spring Boot** | 웹 서버 구축 | 자바 기반 웹앱 만들기 가장 쉬움 |
| **MyBatis** | 데이터베이스 쿼리 | SQL 성능 최적화 가능 |
| **JWT** | 사용자 인증 | 안전한 로그인 시스템 |
| **MariaDB** | 데이터 저장 | 신뢰할 수 있는 데이터 관리 |
| **Swagger** | API 문서화 | 개발자들이 쉽게 사용하도록 문서화 |
| **Caffeine** | 데이터 캐싱 | 같은 데이터 반복 요청 시 빠르게 처리 |

## 🎪 SLIDE 4: 핵심 기능

### 🎯 사용자가 할 수 있는 5가지

#### 1️⃣ 입찰 물건 검색
```
사용자가 "서울 아파트 1억 이상" 검색
  ↓
백엔드가 데이터베이스에서 해당 물건 찾기
  ↓
결과를 화면에 표시
  ↓
"5개 물건을 찾았습니다" 표시
```

#### 2️⃣ 회원가입 & 로그인
```
1. 회원가입: 사용자명, 비밀번호, 이메일 등록
2. 데이터베이스에 안전하게 저장 (비밀번호는 암호화)
3. 로그인 시: JWT 토큰 발급
4. 이 토큰으로 입찰 같은 기능 사용 가능
```

#### 3️⃣ 입찰 신청
```
로그인한 사용자가 물건 선택
  ↓
입찰 금액 입력
  ↓
"입찰 신청" 버튼 클릭
  ↓
데이터베이스에 입찰 기록 저장
  ↓
"입찰이 완료되었습니다!" 메시지 표시
```

#### 4️⃣ 내 입찰 현황 확인
```
로그인한 사용자
  ↓
"내 입찰" 탭에서 입찰 내역 조회
  ↓
본인이 신청한 입찰들만 표시
  ↓
입찰 상태, 금액, 마감일 확인
```

#### 5️⃣ 알림 설정
```
"이 물건이 3일 전에 알려줄래?"
  ↓
user_alerts 테이블에 저장
  ↓
매일 자동으로 확인 (스케줄러)
  ↓
마감 3일 전에 알림 표시
```

## 🎪 SLIDE 6: 성능 최적화

### 🚀 느린 시스템을 빠르게 만드는 기법

#### 1️⃣ 데이터베이스 인덱스

**문제**: "서울 지역 검색"할 때 165개 데이터 전부 확인

```
❌ 인덱스 없음:
   165개 전부 확인 → 1000ms
   (마치 사전을 처음부터 끝까지 읽기)

✅ 인덱스 있음:
   목차 이용해서 바로 찾기 → 100ms (10배 빠름!)
   (마치 사전의 목차로 바로 찾기)
```

#### 2️⃣ 메모리 캐싱 (Caffeine)

```
첫 번째 요청:
  ↓
  DB 조회 → 500ms
  ↓
  결과를 메모리에 저장

두 번째 요청 (10분 내):
  ↓
  메모리에서 바로 조회 → 10ms
  ↓
  10배 빨라짐!

10분 후:
  ↓
  캐시 자동 삭제
  ↓
  새로운 데이터로 갱신
```

#### 3️⃣ 커넥션 풀 (최대 20개 연결)

```
미리 DB 연결 20개 만들어둔다
  ↓
사용자 많아도 빠르게 처리

동시 사용자 처리 능력:
  • 기본 설정: 1명/초
  • 최적화 후: 100명/초
```

#### 성능 개선 결과

| 작업 | 최적화 전 | 최적화 후 | 개선도 |
|------|----------|----------|--------|
| 입찰 검색 | 1000ms | 100ms | 10배 ⬇️ |
| 전체 목록 조회 | 500ms | 50ms | 10배 ⬇️ |
| 동시 사용자 처리 | 1명/초 | 100명/초 | 100배 ⬆️ |

## 🎪 SLIDE 7: 보안 시스템

### 🔐 개인정보 & 입찰 정보를 안전하게

#### 1️⃣ 비밀번호 암호화 (BCrypt)

```
사용자 입력:
  "myPassword123"
    ↓
BCrypt로 암호화:
  "$2a$10$x5K8vK9mL2pQr1sT4uVwZ..."
    ↓
데이터베이스 저장
  (원본은 절대 저장 안 함!)
    ↓
로그인할 때:
  입력 암호화 후 저장된 값과 비교
    ↓
  일치하면 로그인 성공!
```

#### 2️⃣ JWT 토큰 (로그인 상태 유지)

```
로그인 성공
  ↓
토큰 발급:
  "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
    ↓
사용자가 이 토큰을 HTTP 헤더에 담아서 보냄
    ↓
"이 요청이 john123의 요청이 맞는가?" 확인
    ↓
맞으면 요청 처리, 틀리면 401 Unauthorized
```

#### 3️⃣ CORS & Spring Security

- ✓ 허가된 도메인에서만 접근 가능
- ✓ HTTPS 권장 (실제 배포시)
- ✓ SQL Injection 방지 (MyBatis)
- ✓ CSRF 토큰 검증

#### 4️⃣ 데이터 접근 제한

```
✗ 로그인 없이 입찰 불가
✗ 다른 사용자 입찰 내역 못 봄
✓ 자신의 입찰만 조회 가능
✓ 관리자만 전체 데이터 조회 가능
```

## 🎪 SLIDE 8: 개발 과정

### 🛠️ 개발하며 겪었던 3가지 어려움과 해결책

#### 문제 1️⃣: XML 파싱이 어려움

```
상황:
  KAMCO API가 XML 형식으로 데이터 전송
  이 XML을 자바에서 읽고 DB에 저장해야 함
  어떻게 할까?

해결 방법:
  DocumentBuilder 사용해서 단계별 파싱
  ├─ XML 파일 읽기
  ├─ 각 <item> 태그를 하나씩 읽기
  ├─ 필요한 데이터 추출하기
  └─ DB에 저장하기

결과:
  165개 경매 물건 데이터를 성공적으로 로드!
```

#### 문제 2️⃣: JWT 토큰 구현이 복잡

```
상황:
  "로그인 상태를 어떻게 유지할 거야?"
  기존 방식(세션)은 느리다...

해결 방법:
  Spring Security + JWT 라이브러리 사용
  ├─ 로그인 성공 시 토큰 발급
  ├─ 모든 요청에서 토큰 검증하는 필터 추가
  └─ 테스트하며 하나씩 연결

시행착오:
  1. 토큰 발급은 되는데 검증이 안 됨
     → 토큰 형식 점검, 헤더 확인
  2. CORS 오류 발생
     → @CrossOrigin 어노테이션 추가
  3. 프론트에서 토큰을 못 보냄
     → HTTP Authorization 헤더 확인

결과:
  안전한 토큰 기반 인증 시스템 완성!
```

#### 문제 3️⃣: 검색이 느림

```
상황:
  사용자: "왜 검색 결과가 1초나 걸려?"
  개발자: 165개 데이터 전부 스캔 중...

진단:
  1. 데이터베이스 쿼리 확인
     → SELECT * FROM kamco_bids WHERE sido = '서울'
        모든 행을 확인하고 필터링 중!
  
  2. 인덱스 확인
     → sido 컬럼에 인덱스가 없음!

해결 방법:
  1. 지역별 인덱스 추가
     CREATE INDEX idx_sido ON kamco_bids(sido);
  
  2. 자주 조회되는 데이터 캐싱
     전체 입찰 목록을 10분마다 갱신
  
  3. 쿼리 최적화
     필요한 컬럼만 SELECT

성능 개선:
  • 최적화 전: 1000ms
  • 최적화 후: 100ms
  • 개선율: 10배 빨라짐! 🚀
```

### ❓ Q2. "가장 어려웠던 부분은?"

#### ✅ 답변
**JWT 토큰 인증 구현**이었습니다.

**초기 문제점**:
```
로그인 후 토큰을 발급받았는데
입찰 요청을 할 때마다 계속 401 오류가 났어요
```

**디버깅 과정**:
1. 토큰 발급 로직 확인 → 정상
2. HTTP 헤더 확인 → Authorization 헤더 누락
3. Spring Security 필터 설정 확인 → 토큰 검증 로직 재작성
4. 프론트엔드에서 토큰 전송 방식 수정
5. 테스트 계정으로 재테스트 → 성공!

**배운 점**:
- 프론트엔드와 백엔드 간의 통신을 정확히 이해해야 함
- HTTP 헤더의 중요성
- 로그로 중간 과정을 추적하기

```
  □ 큰 목소리로 자신감 있게 발표
  □ 청중과 눈 맞춤
  □ 시간 초과하지 않기 (매 슬라이드 체크)
  □ 어려운 기술용어 쉽게 설명
  □ 질문 환영하는 태도

실습:
  □ 자기 전에 한 번 더 연습
  □ 질문 예상하고 답변 준비
  □ 신경 쓸 필요 없음 (실수는 누구나 함)
```

### 💡 발표 시 강조 포인트

```
1. "이 프로젝트가 왜 필요했는가?"
   → 청중의 공감 얻기

2. "기술적으로 어려웠던 부분과 해결책"
   → 문제 해결 역량 보여주기

3. "성능 최적화 결과" (1000ms → 100ms)
   → 구체적인 수치로 임팩트 주기

4. "아직 구현하지 않은 미래 계획"
   → 성장하고자 하는 마음 보여주기

5. "배운 점" (기술 + 비기술적 배움)
   → 이 프로젝트의 의미 전달
```
